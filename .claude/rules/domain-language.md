# Domain Language

- **vault** = a local Obsidian directory containing `.obsidian/`, `CLAUDE.md`, `raw/`, and `wiki/` subdirectories, registered in `~/.claude.json` for Claude Code MCP access.
- **vault name** = user-chosen identifier matching `^[a-zA-Z0-9_-]+$`, max 64 chars; used as the key in the MCP registry.
- **vault dir** = the full filesystem path to a vault on disk; always resolved via the MCP registry, never from user input.
- **MCP registry** = the `mcpServers` object in `~/.claude.json`, where each vault is registered with its path and expected SHA-256 hash.
- **launcher** = `.mcp-start.js` in each vault root; bytes copied verbatim from `lib/mcp-start.js.tmpl`, pinned to a SHA-256 hash for self-verification.
- **launcher template** = `lib/mcp-start.js.tmpl` — the single source of truth for the launcher. Stays as raw JS (not migrated to TS) because every user vault byte-pins its SHA-256.
- **GitHub Pages** = static site hosting integrated via `src/commands/visibility.ts` and `lib/deploy.yml.tmpl` for publishing `raw/` and `wiki/` content.
- **dispatch** = the flow `vaultkit <cmd>` → `bin/vaultkit.ts` (commander) → `src/commands/<cmd>.ts`. The published package ships compiled `dist/bin/vaultkit.js` → `dist/src/commands/<cmd>.js`.
- **command module** = a `src/commands/<name>.ts` file that exports `async function run(params, options?: <Name>Options): Promise<...>`. Per-command `<Name>Options` interfaces extend `RunOptions` from `src/types.ts`.
- **lib module** = a `src/lib/<name>.ts` file shared by command modules. Current libs: `registry`, `vault`, `vault-templates`, `mcp`, `platform`, `git`, `github`, `errors`, `logger`, `constants`.
- **vault layout files** = the canonical file/directory names that make up a vaultkit vault (`.mcp-start.js`, `CLAUDE.md`, `README.md`, `index.md`, `log.md`, `.gitignore`, `.gitattributes`, `_vault.json`, `.obsidian`, `raw/`, `wiki/`, `.github/workflows/`). Defined in `src/lib/constants.ts` as `VAULT_FILES`, `VAULT_DIRS`, `WORKFLOW_FILES`. Use those constants instead of inline string literals.
- **vault layout** = the operations that build / detect / repair the canonical vault file structure, in `src/lib/vault-layout.ts`. `CANONICAL_LAYOUT_FILES` lists the files in deterministic order; `detectLayoutGaps(vaultDir)` returns the missing subset; `writeLayoutFiles(vaultDir, ctx, files)` writes the named files with content from `renderLayoutFile`. Single source of truth for both `init` (full creation) and `update` (gap repair).
- **Vault class** = the rich object form of a registered vault, defined in `src/lib/vault.ts`. Wraps name + dir + expectedHash and exposes disk/path checks (`existsOnDisk`, `isVaultLike`, `hasGitRepo`, `hasLauncher`, `sha256OfLauncher`). Construct via `Vault.tryFromName(name, cfgPath?)` or `Vault.fromRecord(record)`.
- **vault templates** = the static-content builders in `src/lib/vault-templates.ts` (`renderClaudeMd`, `renderReadme`, `renderIndexMd`, `renderLogMd`, `renderGitignore`, `renderGitattributes`, `renderDuplicateCheckYaml`, `renderVaultJson`). Used by `init` and `update` to scaffold or restore the standard vault layout. Separate from `vault.ts` because they are content, not logic.
- **MCP helpers** = the `claude mcp` CLI wrappers in `src/lib/mcp.ts`: `runMcpAdd`, `runMcpRemove`, `runMcpRepin`, `manualMcpAddCommand`, `manualMcpRemoveCommand`, `manualMcpRepinCommands`, `findOrInstallClaude`. `runMcpAdd` is the single source of truth for the security-critical `--expected-sha256=<hash>` argv shape — every registration flow must go through it. `runMcpRemove` is the matching invariant for removal: `destroy`, `disconnect`, and the `init` rollback path all share one argv shape, so a future `--scope` change is a single edit.
- **Logger** = the structured logging interface in `src/lib/logger.ts` with four levels (`info` / `warn` / `error` / `debug`). `ConsoleLogger` is the production implementation; `SilentLogger` is the test no-op. Replaced the flat `LogFn` type in v2.1.0. Commands receive one via `RunOptions.log`.
- **VaultkitError** = the categorized error class in `src/lib/errors.ts`. Carries a `VaultkitErrorCode` (`INVALID_NAME`, `NOT_REGISTERED`, `ALREADY_REGISTERED`, `NOT_VAULT_LIKE`, `HASH_MISMATCH`, `AUTH_REQUIRED`, `PERMISSION_DENIED`, `TOOL_MISSING`, `NETWORK_TIMEOUT`, `UNRECOGNIZED_INPUT`, `PARTIAL_FAILURE`) that `bin/vaultkit.ts:wrap()` maps to distinct process exit codes (2-12) via the `EXIT_CODES` table. Use `isVaultkitError(err)` for narrowing in `catch` blocks. Plain `Error` continues to exit 1 for unexpected failures.
- **CommandModule** = the lifecycle interface in `src/types.ts` that every `src/commands/<name>.ts` exports satisfies. Generic over `[TParams...]` (positional args as a tuple), `TOptions` (extends `RunOptions`), and `TResult`. Each command file ends with a `_module: CommandModule<...> = { run }` type sentinel that fails the type check if a future command's `run` signature drifts.
- **exit codes** = the public process-exit contract documented in [src/lib/errors.ts](../../src/lib/errors.ts): 0 = success, 1 = unknown error, 2-12 = `VaultkitErrorCode` categories. Scripted callers may rely on these specific codes.

Run `/common-ground` at the start of each session to surface assumptions about this domain.
